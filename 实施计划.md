# video2gif 实施计划(step by step)

> 配套 [技术方案.md](技术方案.md) 的落地计划。原则:**小步可验证、每步都能在真机上看到结果**。
> 关键路径:脚手架 → 导入 → 截取 → 预览(比例/缩放旋转/**拖动**/变速)→ 导出 mp4 →(可选)GIF → 交付。
> 每个阶段给出:**目标 / 步骤 / 关键接触点 / 验收(DoD) / 风险**。验收不过不进下一阶段。

---

## 0. 总体顺序与依赖

```
P0 脚手架/锁版本 ─▶ P1 导入+校验+存储 ─▶ P2 截取 ─▶ P3 预览骨架(Effect 管线)
                                                          │
        ┌─────────────────────────────────────────────────┘
        ▼
P4 比例 ─▶ P5 缩放/旋转(去黑边) ─▶ P6 拖动改变位置 ─▶ P7 变速
        │                                                  │
        └──────────────────────────────────────────────────┘
                                 ▼
            P8 导出 mp4(WYSIWYG 校验)─▶ P9 可选 GIF ─▶ P10 存储交付 ─▶ P11 收尾标定
```

- **P3 是地基**:预览的 Effect 管线骨架必须先通。**P4–P6 是视觉 Effect**(进共用 `buildVideoEffects`);**P7 变速不进 `buildVideoEffects`**——它改帧时间戳,只写 `state.speed`,预览走播放器倍速、导出走 `EditedMediaItem#setSpeed`(§5.4)。
- **P6 拖动**依赖 P4(Crop 已在管线里)、最好在 P5 之后(要处理旋转后内接矩形的余量)。
- **P4–P6 阶段只验预览**(无黑边 / 跟手);**WYSIWYG 对拍统一放 P8**(导出就绪后,对此前所有像素能力跑一次对拍矩阵)。想更早发现问题,可在 P3 后先搭一个**最小导出 harness**(固定 state 一键导出)提前对拍。

## P0. 工程脚手架与版本锁定

**目标**:一个能装能跑的空 app,Media3 全家桶同版本锁死。

**步骤**
1. 新建 Android 工程(单 Activity / Compose 或 View 皆可,先不纠结)。
2. 加依赖并**统一锁版本**:`media3-exoplayer` / `media3-effect` / `media3-transformer` / `media3-common` / `media3-ui`(P3 的 PlayerView)**锁同一个具体版本**。`≥1.10` 只是最低要求(变速 API 依赖,§10.4),Gradle 里要写死验证过的版本——如 **1.10.1**(1.10.0 为 2026-03-26 稳定版,撰写时已到 1.10.1),**不要写动态范围**。
3. 全局准备 `@OptIn(UnstableApi::class)`(多数 Media3 编辑 API 带 `@UnstableApi`)。
4. ffmpeg-kit 先**不引**(留到 P9),避免早期被 `.so` / 16KB / 许可问题拖住。
5. 建空的 `EditState` 数据类(后续各阶段往里加字段)与一个空的 `buildVideoEffects(state): List<Effect>`。

**关键接触点**:`build.gradle`(版本 catalog 锁版本)、`EditState`、`buildVideoEffects`。

**验收(DoD)**
- [ ] app 能装能起,空白页不崩。
- [ ] `./gradlew dependencies` 里 media3 各 artifact 版本一致,且为锁定的具体版本(如 1.10.1,非动态范围)。

**风险**:版本不一致会在 P3 运行期才炸,这里先锁死最省事(§10.4)。

## P1. 导入与校验 + 本地可读路径

**目标**:从相册选一个视频,校验时长 >500ms,拿到一个**确认可读的本地路径** `sourceLocalPath`。

**步骤**
1. 用 `ActivityResultContracts`(PickVisualMedia / GetContent)选视频,拿到 `content://` Uri。
2. `MediaMetadataRetriever` 取总时长;**≤500ms 直接拒绝**并提示。
3. 解析本地路径(**best effort**):尝试从 Uri 拿绝对路径 → **必须** `File.exists() && File.canRead()`(或试开流)验证。
4. 验证不过 / 只有 `content://` → `ContentResolver.openInputStream` **复制一份到 `cacheDir`/`filesDir`**。
5. 记录 `sourceLocalPath`(原文件或副本),后续 ffmpeg 只认它。

**关键接触点**:导入器、`EditState.sourceUri` / `sourceLocalPath` / `durationMs`。

**验收(DoD)**
- [ ] 选一个 >500ms 视频能进入下一步;选 ≤500ms 被拒。
- [ ] `sourceLocalPath` 指向的文件 `exists() && canRead()` 为真。
- [ ] 用一个 scoped-storage 下 `_data` 拿不到的视频测试,能走「复制到 cache」兜底。

**风险**:`_data` 列不可靠;**绝不把 `content://` 直接拼进 ffmpeg**(§5.1 存储)。

## P2. 截取视频(区间)

**目标**:UI 选 `[startMs, endMs]`,约束 `500ms ≤ (end-start) ≤ 10000ms`,默认从 0。

**步骤**
1. 缩略图轨 / 双滑块选区间(可先用两个数字输入跑通逻辑,再美化)。
2. 夹紧约束:不足 500ms 或超 10s 时禁止确认。
3. 区间**只存进 `EditState`**(`clipStartMs/clipEndMs`),导出时才折进 `ClippingConfiguration`(§3);预览阶段可用 ExoPlayer 的 seek/区间循环近似。

**关键接触点**:`EditState.clipStartMs/clipEndMs`、截取 UI。

**验收(DoD)**
- [ ] 选区间被正确夹在 [500ms, 10s]。
- [ ] 区间值正确写入 state,UI 能复现。

**风险**:**逐帧精确**依赖导出走转码路径,不要为了快导出去掉 Effect 走 trim/edit-list 优化(§3)。

## P3. 预览骨架:ExoPlayer + Effect 管线

**目标**:视频能在 PlayerView 播,且 Effect 管线已接通(先只放一个 `Presentation`)。这是后面所有像素能力的地基。

**步骤**
1. 建 ExoPlayer + PlayerView,默认 `MediaCodecVideoRenderer`(别换)。
2. `buildVideoEffects` 先只返回 `Presentation.createForHeight(targetHeight).copyWithUnsetSideRoundedTo(2)`。
3. **在 `prepare()` 之前**至少调一次 `setVideoEffects(buildVideoEffects(state))`(硬约束)。
4. 封装 `applyEffects()`:重建列表 → `setVideoEffects(...)`,供后续手势调用。
5. 截取区间:预览用 seek 到 `clipStartMs` 并到 `clipEndMs` 循环(近似;精确裁剪在导出)。

**关键接触点**:`ExoPlayer`、`PlayerView`、`applyEffects()`、`EditState.targetHeight`。

**验收(DoD)**
- [ ] 视频能播。
- [ ] 确认 effect 管线真的生效:**故意加一个明显滤镜**肉眼验证(验完删)。
- [ ] `targetHeight` 生效不靠肉眼(PlayerView 会缩放显示,看不出):用 P8/最小 harness 导出后看文件 metadata 分辨率,或打调试日志确认输出尺寸。

**风险**:`setVideoEffects` 必须 prepare 前先调一次;不支持改时间戳的 effect(变速别进这里,留 P7);不支持 DRM(§4)。

## P4. 预览 — 比例(中心裁剪,不留黑边)

**目标**:原始 / 1:1 / 3:4 / 4:3 / 16:9 / 9:16 切换,短边贴合长边裁掉,**不留黑边**。

**步骤**
1. `EditState.aspect` 字段 + 比例选择 UI。
2. 写 `centerCropHalfExtents(state)`:按目标比例算中心裁剪的 NDC 半宽/半高 `(halfW, halfH)`(这一步先只含比例,P5 再叠加 scale/旋转)。
3. `cropEffect(state)` 用它返回 `Crop`,插到 `ScaleAndRotate(占位)`→`Crop`→`Presentation` 顺序中。

**关键接触点**:`centerCropHalfExtents`、`cropEffect`、`EditState.aspect`。

**验收(DoD)**
- [ ] 各比例切换即时生效,画面**无黑边**。
- [ ] `Presentation` 输出宽高为偶数(`copyWithUnsetSideRoundedTo(2)`)。

## P5. 预览 — 缩放 / 旋转(去黑边)

**目标**:缩放、旋转(含任意角),旋转后用内接矩形裁掉包围黑矩形。

**步骤**
1. `EditState.scaleX/scaleY/rotation` + pinch 缩放 / 旋转手势 → `applyEffects()`。
2. 几何用 `ScaleAndRotateTransformation`(**只有 scale + rotation**)。
3. 扩展 `centerCropHalfExtents`:把 **scale**(放大→窗口相对更小)与**任意角旋转的内接矩形**(更小)一并算进 `(halfW, halfH)`;90° 步进无黑边问题。
4. **这是全计划最高风险点,要可验证**:给 `centerCropHalfExtents` 写**纯函数单测 + 样例表**,断言 `halfW/halfH ∈ (0,1]` 且**不越界**(裁剪窗口完全落在旋转后内容内 → 无黑边)。最少覆盖下表的笛卡儿积:

   | 比例 | 旋转 | scale | 期望 |
   |---|---|---|---|
   | 原始 / 1:1 / 9:16 | 0° / 30° / 45° / 90° | 1× / 2× | `0 < halfW,halfH ≤ 1`;旋转后窗口在内接矩形内;`maxO = 1-half ≥ 0` |

   - 0°/90° 应等于纯比例中心裁剪(旋转不缩窗口);30°/45° 内接矩形应**明显更小**。
   - 把 P6 的夹紧也一并测:`offset` 推到极限时 `Crop` 四角仍在 `[-1,1]` 内。

**关键接触点**:`ScaleAndRotateTransformation`、`centerCropHalfExtents`(叠加 scale/旋转,**纯函数、带单测**)。

**验收(DoD)**
- [ ] `centerCropHalfExtents` 单测覆盖上表全部组合,断言不越界全绿。
- [ ] 真机放大、旋转 30°/45° 等任意角,**画面四角无黑边**。
- [ ] 90° 步进正常,比例仍正确。

**风险**:ScaleAndRotate 不含平移(平移留 P6);任意角旋转必须内接裁切(§4)。**内接矩形算错 = 黑边/裁过头,务必先让单测兜底再上真机。**

## P6. 预览 — 拖动改变位置(本次新增)

**目标**:在 PlayerView 上拖动,平移取景窗口、**不露黑边**,预览=导出。详见技术方案「拖动改变位置」。

**步骤**
1. `EditState.offsetX/offsetY = 0f`(NDC,取景窗口中心)。
2. `cropEffect` 折进偏移并**夹紧**:`maxO = 1 - half`,`offset.coerceIn(-maxO, maxO)`,`Crop(ox±halfW, oy±halfH)`。
3. PlayerView 挂 `GestureDetector.onScroll`:
   - 像素宽高用 **PlayerView 里「实际视频显示区域」的 rect**(去掉 letterbox/pillarbox 的黑条),**不要**直接用整个 view 尺寸——有黑条时直接用 view 会让拖动比例算错;
   - view 像素 → NDC **乘窗口跨度 `2*half`**(不是 2),保证跟手 1:1;
   - 处理 `onScroll`(旧−新)符号、NDC 的 Y 朝上而 View 朝下 → 实测对齐方向;
   - 旋转后**不用反算方向**(Crop 在 ScaleAndRotate 之后)。
4. **比例/缩放/旋转变化时**:`offset` 也 `coerceIn` 回去并**回写 state**,消除「空拖死区」。
5. 加「双击 / 重置」把 `offset` 归 0。

**关键接触点**:`EditState.offset*`、`cropEffect`(偏移+夹紧)、PlayerView 手势、`applyEffects()`。

**验收(DoD)**
- [ ] 放大后拖动,内容**跟手**且**任何方向都不露黑边**。
- [ ] 无余量构图(原始比例+不放大+无旋转)拖动被夹住(画面不动),符合预期。
- [ ] 切比例/旋转后再拖,无「手指动画面不动」的死区。
- [ ] 双击复位生效。

**风险**:符号/Y 轴要真机校;跟手系数必须用 `2*half`(§4)。

## P7. 预览 — 变速

**目标**:预览倍速,固定档位(如 0.5/1/1.5/2),与导出共用 `state.speed`。

**步骤**
1. `EditState.speed`(校验 `>0`,UI 收敛到固定档)。
2. 预览:`if (isCommandAvailable(COMMAND_SET_SPEED_AND_PITCH)) setPlaybackSpeed(state.speed)`。
3. **变速绝不进 `buildVideoEffects`**(会改时间戳)。
4. 预览音频策略:`setVolume(0f)` 或禁音轨(导出是无音轨,保持一致)。

**关键接触点**:`EditState.speed`、`setPlaybackSpeed`、`setVolume`。

**验收(DoD)**
- [ ] 切档位预览速度变化明显,画面其他效果不受影响。
- [ ] 预览无音频(或按产品定策略),与导出一致。

**风险**:speed 不能接近 0;成品最小时长语义 `选取时长 ÷ speed ≥ 500ms` 待 §10.6 定。

## P8. 导出 mp4(第一个 WYSIWYG 验收点)

**目标**:Transformer 一遍出**无音轨 mp4**,所见即所得。

**步骤**
1. `MediaItem` 折进 `ClippingConfiguration`(P2 的区间)。
2. `EditedMediaItem`:`setRemoveAudio(true)` + `setSpeed(constantSpeedProvider(state.speed))` + `setFrameRate(maxOutputFps)` + `setEffects(Effects(emptyList(), buildVideoEffects(state)))`。
   - `constantSpeedProvider(speed)` 是**自定义的恒定 `SpeedProvider`**(`getSpeed` 恒返回 speed、`getNextSpeedChangeTimeUs` 返回 `C.TIME_UNSET`),实现见技术方案 §5.4。
   - ⚠️ `EditedMediaItem.Builder#setSpeed(SpeedProvider)` 一旦设置,**不能再混用任何 speed-changing effect**;`buildVideoEffects` 里也确认没有改时间戳的 effect(与 P7 一致)。
3. `Transformer`:`setVideoMimeType(H264)` + 码率(`k×W×H×outputFps`,k=低0.05/中0.1/高0.2)。
4. **必须显式 `addListener`** 才有回调;在带 Looper 的线程(主线程)`start()`。
5. 进度:`getProgress(ProgressHolder)` 轮询;UI 提示「导出中勿切后台」。
6. 失败/中断清理 cache 残留。
7. **显式锁定精确裁剪条件**:`Transformer.Builder` 里**显式** `experimentalSetTrimOptimizationEnabled(false)`(别依赖默认值)——防止以后 SDK 默认行为变化或重构把「逐帧精确」的前提悄悄破坏(§3)。

**关键接触点**:`EditedMediaItem.Builder`、`Transformer`(`experimentalSetTrimOptimizationEnabled(false)`)、`constantSpeedProvider`、`DefaultEncoderFactory`。

**验收(DoD)**
- [ ] **像素能力逐项对拍(固定 `speed=1`)**:同一 state 下,导出 mp4 的比例/缩放/旋转/**拖动位置**/分辨率 == 预览,同一时间点截图比对。对拍矩阵**统一用 speed=1**——非 1× 时输出帧与源帧时间不对齐,「同一时间点截图」会被误用(变速另由下一条按时间轴单独验)。
- [ ] **变速单独验**(不是同帧截图,而是验时间轴):① 输出时长 = 选取时长 ÷ speed;② 播放速度肉眼/逐帧符合 speed;③ **抽帧时间映射**正确——输出帧 `outputTimeMs` 对应源时间 ≈ `clipStartMs + outputTimeMs × speed`(抽几个关键帧核对源画面)。
- [ ] 竖屏源导出后播放方向正常(注意宽高交换特例)。
- [ ] 体积 ≈ `bitrate×输出时长/8`,输出时长 = 选取时长 ÷ speed。

**风险**:别走 trim optimization / edit-list(逐帧精确条件,已在步骤 7 显式 `experimentalSetTrimOptimizationEnabled(false)` 兜底);竖屏 encoded 维度可能被交换(§5.1)。

## P9. (可选)mp4 → GIF

**目标**:对刚导出的 mp4 跑一条 ffmpeg-kit 命令转 GIF,**不做 scale**。

**步骤**
1. 引入 ffmpeg-kit 社区 fork(`...-16kb`),**锁版本、校验来源、记许可(LGPL/GPL)**。
2. `executeWithArgumentsAsync(arrayOf("-i", mp4, "-vf", filterGraph, outGif), cb)`——**参数数组、不带 `ffmpeg` 前缀**。
3. filtergraph 只调 `fps + palettegen/paletteuse + dither`(档位:低 8/128/`none`、中 12/256/`bayer`、高 15/256/`sierra2_4a`);**`dither=none` 不是 `off`**。
4. statistics 回调出进度。

**关键接触点**:ffmpeg-kit 调用、filtergraph 档位表。

**验收(DoD)**
- [ ] GIF 能生成、方向正确(尤其竖屏,实测 FFmpeg 处理旋转元数据)。
- [ ] 三档画质/体积差异符合预期,分辨率与 mp4 一致(FFmpeg 没 scale)。

**风险**:ffmpeg-kit 非「零 native」、官方已退休、许可合规(§5.2 / §8)。

## 全局验收法:WYSIWYG 对拍

**P4–P6 阶段**:只验**预览**(无黑边 / 跟手 / 各比例正确),**不要求**此时就能导出。

**P8 阶段(导出就绪)**:对此前所有像素能力跑一次**对拍矩阵**——
1. 在预览里把各能力组合到明显值(大旋转 + 放大 + 拖到角落 + 某比例)。
2. 用**同一 `state`** 导出 mp4。
3. 截预览帧 vs mp4 同位置帧,**像素级目测一致**。

> 想在 P4–P6 就提前对拍:P3 后搭一个**最小导出 harness**(写死 state、一键 `Transformer.start`),不必等 P8 完整导出 UI。
>
> 像素变换的真值只在 `buildVideoEffects`;只要预览和导出都调它,理论上必然一致。对不上 → 先查是不是有能力没进 / 误进 `buildVideoEffects`(例如把变速塞了进去)。
