# video2gif 技术方案

> Android 原生 GIF / 短视频制作工具的技术设计文档。
> 核心结论:**以 Media3（`androidx.media3`）的 Effect 管线处理视觉变换**,预览与导出共用同一条视觉效果链(WYSIWYG);变速是时间轴变换,单独走 `state.speed`。
> **导出主交付物是无音轨 mp4;GIF 是可选的第二步**——对已导出的 mp4 跑一条 ffmpeg 命令转换。

---

## 1. 总体架构

```
┌─────────────────────────────────────────────────────────────┐
│                     编辑状态 EditState                         │
│  裁剪区间 / 比例 / 缩放旋转拖动 / 变速 / 滤镜 / 文字 / 贴纸 / PAG │
└─────────────────────────────────────────────────────────────┘
            │ buildVideoEffects(state)  ← 视觉效果真值(不含变速)
            ▼
   ┌──────────────────────┐        ┌──────────────────────────┐
   │  预览态                │        │  导出态                    │
   │  ExoPlayer            │        │  Transformer              │
   │  .setVideoEffects()   │        │  .start(edited, out)      │
   └──────────────────────┘        └──────────────────────────┘
        同一组视觉 Effect + 同一 speed → 预览 == 导出(WYSIWYG)
                                         │
                                         ▼  最终 mp4(交付物)
                                  ┌──────────────────────┐
                                  │ 可选:转 GIF           │
                                  │ ffmpeg 单命令(mp4→gif)│
                                  └──────────────────────┘
```

**核心原则:像素变换(裁剪 / 缩放旋转拖动 / 滤镜 / 文字 / 贴纸)的真值只存在于 Media3 Effect 层。** 预览只显示,导出用同一组 Effect 把像素 bake 出来。变速不属于像素变换,是时间轴变换,不进入共用的 `buildVideoEffects`。整条 Effect 管线跑在 **OpenGL ES** 上,自定义 GL 滤镜是一等公民。

---

## 2. 功能流程

| 步骤 | 功能 | 技术方案 |
|---|---|---|
| 1 | 从相册导入单个视频(时长 > 500ms) | `ActivityResultContracts` 选视频 + `MediaMetadataRetriever` 校验时长 |
| 2 | 截取视频(x→y ms,最短 500ms 硬约束;**10s 上限为软约束**,默认从 0) | 记录区间,折进导出的 Transformer pass(`ClippingConfiguration`) |
| 3 | 预览(比例 / 缩放 / 旋转 / 拖动 / 变速) | ExoPlayer + Media3 Effect 管线;变速走播放器倍速(见 §5.4) |
| 4 | 导出 | Media3 Transformer 出 **mp4**(`EditedMediaItem.Builder#setSpeed` 含变速);可选 ffmpeg 命令转 GIF |

---

## 3. 第 2 步:截取视频

### 决策:用 Transformer 的 `ClippingConfiguration`,不手写 MediaCodec

- 变换放在 Effect 层、自定义 GLES shader FFmpeg 跑不了,所以导出时**必然有一个 Transformer pass 把效果 bake 进像素**,裁剪直接塞进这个 pass,仅几行。
- 手写 MediaCodec(InputSurface/EGL/编解码循环/EOS/时间戳)是几百行且各芯片行为不一,还多一次编码,纯亏。

```kotlin
val item = MediaItem.Builder()
    .setUri(srcUri)
    .setClippingConfiguration(
        ClippingConfiguration.Builder()
            .setStartPositionMs(startMs)
            .setEndPositionMs(endMs)
            .build())
    .build()
```

### ⚠️ "逐帧精确"是有条件的

- **只有在强制转码 / 效果 bake 路径下才稳定逐帧精确**——本方案应始终保留非 no-op Effect / 强制转码路径,确保效果真正 bake 进像素。
- 反例:若未来出现"无任何 Effect"的导出、走 trim optimization 或 MP4 edit list 裁剪,**播放器可能从前一个关键帧开始显示**,且 edit-list 裁剪本身有兼容性限制(见 Transformer 文档)。
- 结论:**逐帧精确成立的条件是——不启用 trim optimization、不走 MP4 edit-list,且实际确认走了转码路径**。不要为了"快导出"去掉 Effect 走 edit-list 优化。

> 校验:导入时 `MediaMetadataRetriever` 取总时长拒绝 ≤500ms,并顺带读显示宽高+旋转算 `sourceAspectRatio`(截取页按比例定宽、无黑边)。截取 UI **硬夹最短 500ms**;**10s 为软上限**——滑动允许超过,点「下一步」时若 >10s 才提示拦下(`exceedsMaxClip`)。

## 4. 第 3 步:预览

### 方案:ExoPlayer + Media3 Effect 管线

```kotlin
fun buildVideoEffects(state: EditState): List<Effect> = buildList {
    // 1) 色彩滤镜(先做,作用于原始像素)
    addAll(state.filters.flatMap { it.toEffects() })
    // 2) 几何变换
    add(ScaleAndRotateTransformation.Builder()
        .setRotationDegrees(state.rotation)
        .setScale(state.scaleX, state.scaleY)
        .build())
    add(cropEffect(state))        // 动态 Crop:承担"比例裁剪 + 拖动平移 + 旋转后去黑边"
    // 3) 最终像素尺寸,唯一真值。createForHeight 按比例推导宽度,可能得到奇数宽。
    //    注:copyWithUnsetSideRoundedTo(2) 在锁定的 1.10.1 未确认存在 → 实现里只用 createForHeight,
    //    偶数维度由偶数 targetHeight + 导出端 encoder 兜底;若需强制偶数可改 createForWidthAndHeight。
    add(Presentation.createForHeight(state.targetHeight))
    // 4) 叠加层(最后,在输出坐标系)
    add(OverlayEffect(buildOverlays(state)))
}
```

### 各能力的准确归属(重要修正)

| 能力 | 用什么 | 说明 |
|---|---|---|
| 缩放 / 旋转 | `ScaleAndRotateTransformation` | **该 API 只有 scale + rotation** |
| **拖动 / 平移** | **动态 `Crop` 窗口偏移**(`state.offset`)+ 夹紧防黑边 | ScaleAndRotate **不**含平移;详见下"拖动改变位置" |
| 比例(原始/1:1/3:4/4:3/16:9/9:16) | `Crop` 中心裁剪,**不留黑边** | 短边贴合、长边裁掉 |
| **任意角度旋转去黑边** | 旋转后再 `Crop` 内接矩形 | 任意角旋转会产生**包围黑矩形**,与"不留黑边"冲突,必须旋转后裁切;90° 步进无此问题 |
| 最终分辨率 | `Presentation`(按分辨率或比例缩放的 effect) | **像素尺寸的唯一真值,FFmpeg 不再 scale** |

### ⚠️ 预览(`ExoPlayer.setVideoEffects()`)的约束

- **必须在 `prepare()` 之前至少调用一次**来建立效果管线;之后可再调更新。
- 需要 **所有 Media3 artifact 同版本**运行时依赖(`media3-exoplayer` / `media3-effect` / `media3-transformer` / `media3-common` 等)。
- 依赖默认的 **`MediaCodecVideoRenderer`**(不要换成不支持 effect 的 renderer)。
- **不支持会改变帧时间戳的 effect**(→ 变速不能进预览的 effect 列表,必须用播放器倍速,见 §5.4)。
- **不支持 DRM 保护内容**。
- 手势更新 `EditState` → 重新 `buildVideoEffects` → `setVideoEffects(...)` 刷新预览(拖动 / 缩放 / 旋转 / 比例都走这条;拖动细节见下)。

### 拖动改变位置:预览手势 → `Crop` 偏移(WYSIWYG)

**真值落点**:拖动只产生一个**归一化平移量** `state.offsetX / offsetY`,由 `cropEffect(state)` 折进 `Crop` 的窗口中心。因为它进的是**共用的 `buildVideoEffects`**,预览拖到哪、导出就 bake 到哪——**导出侧零改动**(`EditedMediaItem` 仍只调 `buildVideoEffects(state)`,§5.1)。

**为什么是平移"裁剪窗口"而非平移"图层"**:本方案"**不留黑边**"。把取景窗口在源帧上平移再夹紧到内容边界,天然不会露黑;平移图层(`MatrixTransformation` translate)会在另一侧露黑,得靠放大兜底,语义更绕。所以拖动 = **在源帧上挪动裁剪窗口(reframe)**,不是把视频当贴纸挪。

```kotlin
data class EditState(
    // ... 比例 / scale / rotation ...
    val offsetX: Float = 0f,   // 取景窗口中心,NDC([-1,1]),0 = 居中
    val offsetY: Float = 0f,
)
```

`cropEffect` 折进平移 + 夹紧(防黑边):

```kotlin
fun cropEffect(state: EditState): Crop {
    // 目标比例居中裁剪后的半宽 / 半高(NDC,≤1)
    // ⚠️ 这里要把【比例 + 缩放 scale + 任意角旋转的内接矩形】一并算进去:
    //    scale 越大 → 窗口相对越小 → 可平移余量越大;任意角旋转用内接矩形(更小)
    val (halfW, halfH) = centerCropHalfExtents(state)

    // 可平移余量:窗口必须整体留在有效内容里 → |中心| ≤ 1 - 半边
    val maxOx = (1f - halfW).coerceAtLeast(0f)
    val maxOy = (1f - halfH).coerceAtLeast(0f)
    val ox = state.offsetX.coerceIn(-maxOx, maxOx)
    val oy = state.offsetY.coerceIn(-maxOy, maxOy)

    // Crop(left, right, bottom, top),NDC,(0,0) 居中
    return Crop(ox - halfW, ox + halfW, oy - halfH, oy + halfH)
}
```

手势(挂在 PlayerView 上):

```kotlin
GestureDetector(context, object : GestureDetector.SimpleOnGestureListener() {
    override fun onScroll(e1: MotionEvent?, e2: MotionEvent, dxPx: Float, dyPx: Float): Boolean {
        val rect = displayedVideoRect()              // PlayerView 内实际画面矩形(去 letterbox)
        val (halfW, halfH) = centerCropHalfExtents(state)
        // ⚠️ PlayerView 显示的是【裁剪窗口内】的内容,不是整幅源帧:
        //    view 像素 ↔ NDC 要乘窗口 NDC 跨度 2*half,才能"内容跟手"(放大后也不飘)
        val ndcDx = (dxPx / rect.width())  * (2f * halfW)
        val ndcDy = (dyPx / rect.height()) * (2f * halfH)
        state = state.copy(
            // onScroll 的 dx/dy 是(旧−新);NDC 的 Y 朝上、View 的 Y 朝下 → 符号以实测"跟手"为准
            offsetX = state.offsetX + ndcDx,
            offsetY = state.offsetY - ndcDy,
        )
        applyEffects()                                // 重新 buildVideoEffects → setVideoEffects(...)
        return true
    }
})
```

要点:

- **跟手 1:1**:换算乘 `2*half`(窗口跨度),不是乘 2(整帧跨度);否则放大后拖动会"飘"。
- **旋转后不用反算手势方向**:`Crop` 排在 `ScaleAndRotate` **之后**,裁剪坐标系 = 已旋转的输出/屏幕坐标系,屏幕横向拖动就是裁剪横向。只需校准 `onScroll` 的(旧−新)符号与 Y 轴朝向。
- **夹紧即真值,且要回写**:`cropEffect` 每次 build 都重夹一遍;**比例 / 缩放 / 旋转一变,余量就变**,要把 `state.offset` 也 `coerceIn` 回去并**回写 state**,否则会出现"手指在动、画面不动"的空拖死区。
- **有余量才拖得动**:`halfW = halfH = 1`(原始比例 + 不放大 + 无旋转)时 `maxOx = maxOy = 0`,平移被夹成 0——正确,因为任何平移都会露黑。UI 可据此提示或禁用,或让它自然夹住。
- **复位**:提供"双击 / 重置"把 `offset` 归 0。
- **与变速正交**:平移是像素变换、走 `buildVideoEffects`;变速仍单独走 `state.speed`(§5.4),互不影响。

> 备选语义:若产品其实想要"画布上自由摆放、允许背景 / 填充",那是平移**图层**而非裁剪窗口,需引入背景色 / 模糊填充并**放弃"不留黑边"**。当前按**不留黑边的 reframe** 设计;若要前者,请在 §10 标注切换。

## 5. 第 4 步:导出

### 5.1 主交付物:无音轨 mp4(Media3 Transformer 一遍)

```kotlin
val speedProvider = constantSpeedProvider(state.speed)

val edited = EditedMediaItem.Builder(item)         // item 含 ClippingConfiguration
    .setRemoveAudio(true)
    .setSpeed(speedProvider)                       // 变速走 EditedMediaItem,不要塞进 videoEffects
    .setFrameRate(state.maxOutputFps)              // 加速时封顶有效输出 fps,避免超高帧率
    .setEffects(Effects(emptyList(), buildVideoEffects(state)))
    .build()

val transformer = Transformer.Builder(context)
    .setVideoMimeType(MimeTypes.VIDEO_H264)
    .setEncoderFactory(DefaultEncoderFactory.Builder(context)
        .setRequestedVideoEncoderSettings(
            VideoEncoderSettings.Builder().setBitrate(bitrateForTier).build())
        .build())
    .addListener(object : Transformer.Listener {           // ← 必须显式注册才有回调
        override fun onCompleted(c: Composition, r: ExportResult) { /* 成功:落 cache → 导出 */ }
        override fun onError(c: Composition, r: ExportResult, e: ExportException) { /* 失败清理 */ }
    })
    .build()

transformer.start(edited, outMp4Path)   // outMp4Path 用真实文件 path,见下"存储"
```

- **分辨率**:`Presentation` 设目标高度 240/360/480/540/720/1080(**像素尺寸唯一真值**)。
- **清晰度**:码率 `bitrate ≈ k × W × H × outputFps`,k = 低 0.05 / 中 0.1 / 高 0.2 bpp。变速后用封顶后的 `outputFps` 估算。
- **文件大小预估**:`size ≈ bitrate × 秒数 / 8`(近似精确)。
- 提示用户"分辨率和清晰度越高,文件越大"。

#### ⚠️ 竖屏宽高交换特例
Transformer 转码竖屏(高 > 宽)输出时,**可能交换 encoded frame 的宽高**以提升编码器兼容性,靠 rotation/矩阵元数据还原显示。播放通常正常,但 **GIF 二次转码要实测 FFmpeg 是否正确处理该旋转/矩阵元数据**,否则 GIF 可能横躺。所以"分辨率唯一真值"是指**视觉尺寸**,encoded 维度可能被交换。

#### ⚠️ 存储:优先拿绝对路径,拿不到再复制本地
Transformer 可以直接吃 `MediaItem` 的 `Uri`,但 ffmpeg-kit 命令最稳的是**真实文件路径**。导入视频时按这个顺序处理:
1. 用户选中相册视频后,**尽力(best effort)**从 `Uri` 解析绝对文件路径。**这只是尽力而为**:`_data` 列在 scoped storage 下不保证可用,拿到字符串后**必须验证文件存在且当前进程真能读**(`File.exists()` + `File.canRead()` / 试开流),不能假定一定有效。
2. 解析不到、验证不通过、或只拿到 `content://` 时,用 `ContentResolver.openInputStream(uri)` 复制一份到 **app `cacheDir` / `filesDir`**。
3. 后续统一记录 `sourceLocalPath`:若拿到绝对路径就指向原文件;否则指向复制出来的本地副本。
4. 导出 mp4 先落到 **app `cacheDir` / `filesDir`**(自有路径,直接给 path)。
5. (可选)GIF:对导出的 mp4 跑 ffmpeg-kit 命令,同样输出到 cache。
6. 完成后再**复制到 `MediaStore` / 分享目标**。

关键规则:**不要把 `content://` 直接拼进 ffmpeg 命令**。能拿到可读绝对路径就少一次 IO;拿不到就复制本地,换取稳定性。

### 5.2 可选第二步:mp4 → GIF(一条 ffmpeg 命令)

用户点"转为 GIF"时,对**刚导出的最终 mp4**跑一条命令(经 ffmpeg-kit 执行):

```bash
# shell 示例(带 ffmpeg 前缀,便于阅读)
ffmpeg -i final.mp4 -vf \
"fps=12,split[s0][s1];\
[s0]palettegen=max_colors=256:stats_mode=diff[p];\
[s1][p]paletteuse=dither=sierra2_4a" out.gif
```

> ⚠️ 实际调 FFmpegKit 时**不要带 `ffmpeg` 前缀**。优先用参数数组 API,避免 path / filtergraph 里的空格被命令字符串拆错:
> `FFmpegKit.executeWithArgumentsAsync(arrayOf("-i", finalMp4Path, "-vf", filterGraph, outGifPath), callback)`

- **FFmpeg 只做 fps + palettegen/paletteuse,不做 scale**——mp4 已经是目标分辨率(`Presentation` 锁定),分辨率真值不在 FFmpeg。
- **清晰度档**只调 fps + max_colors + dither(注意 `dither` 合法值):

  | 档 | fps | max_colors | dither |
  |---|---|---|---|
  | 低 | 8 | 128 | `none`(关闭抖动) |
  | 中 | 12 | 256 | `bayer`(有序抖动) |
  | 高 | 15 | 256 | `sierra2_4a` / `floyd_steinberg`(误差扩散) |

  > ⚠️ `dither=off` **不是合法值**,关闭抖动是 `dither=none`。
- **取舍**:GIF 画质继承所转 mp4 的画质(若要最佳 GIF,先导出高码率 mp4 再转)。
- 多一步、但简单:无 libav*、无 C/C++、无中间临时文件、无 GL 一致性问题(mp4 像素已最终化)。

> **依赖性质要标清楚**:ffmpeg-kit **不是"零 native 代码"**——你不写 C/C++,但 AAR 仍带 `.so`,仍有 16KB page / ABI / 包体 / LGPL-GPL 合规问题。且**官方 FFmpegKit 已退休**,`com.moizhassan.ffmpeg:ffmpeg-kit-16kb` 是**第三方社区 fork / Maven build**:默认 **LGPLv3**,启用 GPL 时变 **GPLv3**。→ **锁版本、校验来源**,不可当稳态官方依赖。

### 5.3 后台执行

- **不引入前台 Service、不引入 WorkManager**,就用各自的 async API。
- **导出本身已是异步,不要手动再包一层裸 `Thread`**:
  - `Transformer` 内部用自己的线程转码,但**必须在带 `Looper` 的线程(通常主线程)创建并 `start()`**,`Transformer.Listener` 回调也回到该线程——别丢到无 Looper 的 IO 线程。
  - GIF 那步用 `FFmpegKit.executeWithArgumentsAsync(...)`,同样自带后台线程 + 回调,直接在主线程发起即可。
- 进度:`Transformer` 用 `getProgress(ProgressHolder)` 轮询;ffmpeg-kit 用 statistics 回调。
- 导出时**提示用户"导出中,请勿切到后台"**(切后台 / 锁屏可能被系统回收,导出中断)。
- 导出页保持前台、给进度条;中断 / 失败时清理 cache 残留文件。

### 5.4 变速(预览 + 导出)

变速是**唯一一个预览与导出走不同机制**的功能——它会**改帧时间戳**,而 `ExoPlayer.setVideoEffects()` 明确**不支持改时间戳的 effect**(见 §4)。因此分两条路,但用**同一个 `state.speed` 值**保证一致:

| 路径 | 机制 | 代码 |
|---|---|---|
| 预览 | 播放器级倍速(**非** effect) | `exoPlayer.setPlaybackSpeed(state.speed)` |
| 导出 | `EditedMediaItem.Builder#setSpeed(SpeedProvider)` | 见下 |

**关键:不要把变速放进 `buildVideoEffects`(那是预览/导出共用的视觉效果链)。`SpeedChangeEffect` 已废弃,导出用 `EditedMediaItem.Builder#setSpeed(SpeedProvider)`。**

```kotlin
// 预览:变速走播放器,不进 effect 列表
if (exoPlayer.isCommandAvailable(Player.COMMAND_SET_SPEED_AND_PITCH)) {
    exoPlayer.setPlaybackSpeed(state.speed)
}
exoPlayer.setVideoEffects(buildVideoEffects(state))      // 不含变速

// 导出:变速走 EditedMediaItem#setSpeed(无音轨,免去音频变速/变调处理)
val edited = EditedMediaItem.Builder(item)
    .setRemoveAudio(true)
    .setSpeed(constantSpeedProvider(state.speed))
    .setFrameRate(state.maxOutputFps)          // state.speed > 1 时尤其重要
    .setEffects(Effects(emptyList(), buildVideoEffects(state)))
    .build()

fun constantSpeedProvider(speed: Float): SpeedProvider = object : SpeedProvider {
    override fun getSpeed(timeUs: Long): Float = speed
    // 恒定速度=无后续变化点
    override fun getNextSpeedChangeTimeUs(timeUs: Long): Long = C.TIME_UNSET
}
```

> ⚠️ 该套变速 API 需 **Media3 ≥ 1.10**(`setSpeed` 自 1.9.0、`setFrameRate` 自 1.10),见 §10.4。

注意点:
- **速度必须校验**:`state.speed` 必须 > 0,UI 建议先收敛到固定档位(如 0.5 / 1 / 1.5 / 2)或明确范围,避免接近 0 导致输出时长和体积爆炸。
- **预览音频要定策略**:导出是无音轨 mp4;若产品预览也按无音频处理,预览时要 `exoPlayer.setVolume(0f)` 或禁用音轨,避免源音频跟着 `setPlaybackSpeed` 变速播放,造成预览与导出不一致。
- **时长 / 体积随速度变**:输出时长 = 选取时长 ÷ speed。0.5× 慢放让 10s 选段变 20s → GIF 体积翻倍。**文件大小预估的"秒数"要用输出时长**。
- **成品最小时长也要定语义**:若产品要求最终 mp4/GIF 仍 ≥500ms,加速后也要校验 `选取时长 ÷ speed ≥ 500ms`;否则 500ms 源片段在 2× 下会变 250ms。
- **加速需要帧率封顶**:`setSpeed` 会调整 sample timestamps,加速后有效输出 fps 可能升高;用 `setFrameRate(maxOutputFps)` 防止主交付 mp4 出现过高帧率,码率/体积估算也用封顶后的 fps。
- **10s 上限语义**:截取的 ≤10s 限制建议仍作用在**源选取**;若允许慢放,需决定是否再对**输出时长**设上限,避免超大 GIF(→ §10 待确认)。
- **慢放画质**:Media3 **不做帧插值**,慢放是拉长时间戳,源 fps 低会顿;丝滑慢动作需高帧率源或额外插帧方案(超出当前范围)。
- **动态贴纸 / PAG 的时间轴要定语义**:默认建议动画跟最终输出时间走;如果某类素材必须绑定源片段时间,需要显式用 `sourceTimeUs = outputTimeUs * speed` 做映射并单测。
- **对 GIF 那步基本透明**:GIF 的 `fps=` 会对已导出的 mp4 重采样;但 mp4 本身仍要先通过 `setFrameRate` 控住有效输出 fps。
- 当前仅需**恒定倍速**;分段变速后续直接扩展 `SpeedProvider`。

## 6. 效果链:滤镜 / 文字 / 贴纸 / View / PAG

**除变速这种时间轴变换外,视觉能力都是 Effect**,加视觉功能 = 往列表里塞一个 Effect。预览导出共用,对 FFmpeg 透明(GIF 转换时只看到已合成像素)。

### 6.1 滤镜
- 内置:`RgbAdjustment` / `HslAdjustment` / `RgbMatrix` / `SingleColorLut`。
- 自定义:`GlEffect` / `GlShaderProgram` 写任意 GLSL。

### 6.2 文字 / 贴纸(`OverlayEffect`)
- `TextOverlay`(文字)、`BitmapOverlay`(贴纸)。
- 位置/缩放/旋转/透明度经 `OverlaySettings`;支持随时间变化(`getOverlaySettings(timeUs)`)做动效。

### 6.3 原生 View / PAG(`TextureOverlay`)

统一入口 `TextureOverlay` 的契约:**每帧给一个 GL 纹理**。View、PAG、Lottie、画中画都从这里进。

- **原生 View**:静态 → `drawToBitmap()` → `BitmapOverlay`;动态 → 逐帧 Bitmap(慢)或渲染到 GL 纹理(快)。编辑时可在 PlayerView 上盖真实可交互 View,导出走 overlay 烧入,二者同一状态驱动。

- **PAG(libpag)**:`PAGSurface.FromTexture` 绑定 GL 纹理,`PAGPlayer.setProgress(p)+flush()` 渲染;包进 `TextureOverlay`,把 `presentationTimeUs` 映射成 progress。

```kotlin
class PagOverlay(val pagFile: PAGFile, val durationUs: Long) : TextureOverlay() {
    override fun getTextureId(timeUs: Long): Int {
        player.progress = (timeUs.toDouble() / durationUs).coerceIn(0.0, 1.0)
        player.flush()
        return texId
    }
    override fun getTextureSize(timeUs: Long) = Size(w, h)
    override fun getOverlaySettings(timeUs: Long) = settings
}
```

#### ⚠️ PAG 生命周期与上下文约束
- `PAGSurface.FromTexture` 要求:**当前有 OpenGL context**、纹理为 **`GL_TEXTURE_2D`**、**纹理未绑定到 framebuffer**;不满足会**返回 null**。
- 纹理与 PAGSurface/PAGPlayer 必须在 **Media3 GL 线程**上、于 `TextureOverlay` 的 **configure 建、release 释放**,生命周期严格绑定。
- **GL context 共享**(libpag 与 Media3 effect 管线)是整件事最易卡的点。

> **时间映射是优点**:Media3 按输出帧逐帧回调 overlay,动画确定性烧入,不依赖实时性能——导出慢也不丢帧。

## 7. 可扩展架构:图层模型

```kotlin
sealed interface EditLayer { fun toEffects(): List<Effect> }

data class ColorFilter(...)   : EditLayer   // SingleColorLut / RgbAdjustment
data class ShaderFilter(...)  : EditLayer   // GlEffect 自定义滤镜
data class TextLayer(...)     : EditLayer   // TextOverlay
data class StickerLayer(...)  : EditLayer   // BitmapOverlay
data class PagLayer(...)      : EditLayer   // TextureOverlay
```

**顺序真值(见 §4)**:色彩滤镜 → 几何(ScaleAndRotate → Crop → Presentation)→ 叠加层最后。加视觉功能 = 新增 `EditLayer` 子类 + `toEffects()`,预览/导出代码不动;变速单独由 `state.speed` 驱动。

## 8. GIF 转码:ffmpeg-kit 命令行(选定)与 自建 libav*(备选)

### 选定:ffmpeg-kit 预编译命令行(第三方社区包)
- 不写 C/C++,预编译包已含 mov/mp4 demuxer、file protocol、parser、swscale、avutil 等完整依赖,能直接打开 Transformer 生成的 mp4。
- 但**不是"零 native"**:AAR 带 `.so`,仍有 16KB page / ABI / 包体 / 许可问题。
- **官方 FFmpegKit 已退休**,`com.moizhassan.ffmpeg:ffmpeg-kit-16kb` 是社区 fork(默认 LGPLv3,启 GPL 变 GPLv3)→ **锁版本、校验来源、定期评估维护风险**;长期可考虑自建(见备选)。
- 选 16KB page 对齐变体(`...-16kb`),适配 Android 15+ / Play。

### 备选:自建 libav* + C/C++(若日后要极致控制 / 包体)
若改走自编,需注意(这些是自编路线的坑,选命令行则无需关心):

1. **组件不能只裁 decoder/filter/muxer**:读 Transformer 生成的 mp4 至少还要 **mov/mp4 demuxer、file protocol、对应 parser**,以及 **swscale / avutil** 等依赖,否则会配出"打不开中间 mp4"的包。
2. **16KB page 对齐不止一行**:NDK **r28+ 默认支持**;**r27 及以下**还要加 `-Wl,-z,common-page-size=16384`,且**所有 prebuilt .so 也要重编 / 替换**。
3. **LGPL 合规:`--disable-gpl` 是必要但不充分**。FFmpeg 官方 LGPL checklist 还要求:**分发对应源码、构建说明、许可 / 归属声明、EULA 不得禁止逆向**等;动态链接本身**不能**概括为"已满足"。
4. 用 `avfilter_graph_parse_ptr()` 直接吃命令行同款滤镜图字符串,量化逻辑不用自己写。

## 9. 关键决策记录

| 决策 | 选择 | 理由 |
|---|---|---|
| 第 2 步裁剪 | Transformer `ClippingConfiguration` | 导出 pass 跑不掉,折进去仅几行;逐帧精确需禁用 trim/edit-list 优化并确认走转码 |
| 像素变换位置 | Media3 Effect 层(真值) | 预览导出共用,WYSIWYG |
| 平移 / 拖动 | 动态 `Crop` 窗口偏移(`state.offset`)+ 夹紧防黑边 | 不留黑边的 reframe;进共用 `buildVideoEffects` → 导出零改动 |
| 任意角旋转 | 旋转后再 Crop 内接矩形 | 去包围黑矩形,满足"不留黑边" |
| 变速 | 预览 `setPlaybackSpeed`,导出 `EditedMediaItem.Builder#setSpeed(SpeedProvider)` | 改时间戳的 effect 不能进预览,且 `SpeedChangeEffect` 已废弃 |
| 分辨率真值 | `Presentation`(FFmpeg 不 scale) | 单一真值,避免冲突 |
| 导出格式 | mp4 主交付 + 可选 ffmpeg 命令转 GIF | 简单:无 libav*/C++/中间临时文件 |
| GIF 转码 | ffmpeg-kit 命令行(第三方社区 fork) | 不写 C/C++、依赖完整;但带 .so、需锁版本/合规 |
| 存储 | 优先取可读绝对路径;拿不到再复制到 cache/filesDir | 用真实 path,不直接拼 content:// |
| 后台 | 普通异步线程 + 提示勿切后台 | 不上前台 Service / WorkManager,最简 |
| View / PAG | `TextureOverlay` 统一接入 | 预览导出一致、对 FFmpeg 透明 |

## 10. 待确认 / 后续

1. **输入格式范围**:仅 H.264,还是含 HEVC / 竖屏 rotation metadata / 可变帧率(VFR)?影响 Transformer 旋转处理与解码兼容。
2. **清晰度三档**确切参数标定表(实测校准 k / fps / colors)。
3. **PAG ↔ Media3 GL context 共享 + configure/release 生命周期**的最小骨架(最易卡处)。
4. **API 版本锁定**:**所有 Media3 artifact 必须同一版本,且 ≥ 1.10**(变速依赖:`EditedMediaItem.Builder#setSpeed(SpeedProvider)` 自 1.9.0、`setFrameRate()` 自 1.10;旧的 `Effects#createExperimentalSpeedChangingEffect()` / `SpeedChangeEffect` 已弃用)。统一锁 `media3-exoplayer`/`media3-effect`/`media3-transformer`/`media3-common` 等;这些 API 多数带 `@UnstableApi`,Kotlin 代码需显式 `@OptIn(UnstableApi::class)`;libpag、ffmpeg-kit 版本固定后核对 API 名称。
5. **HDR 源视频**:相册可能有 HDR(HLG/PQ,10-bit)视频。导出到 8-bit mp4/GIF 需色调映射,自定义 `GlShaderProgram` 要处理 `useHdr` 分支,否则颜色发灰/偏色。需决定:统一 tone map 到 SDR,还是检测到 HDR 时降级处理。
6. **变速范围与输出上限**:确认支持哪些固定倍速档,以及慢放后是否限制输出时长、加速后 `maxOutputFps` 取多少。
7. **变速下动态 Overlay/PAG 时间轴**:确认动画跟最终输出时间还是源片段时间走,并用 0.5× / 2× 样例验收。
8. **拖动改变位置的 UX 语义**:① 默认"不留黑边的 reframe",还是要"自由摆放 + 背景/填充"(§4 备选语义);② 拖动是否与 pinch 缩放联动成一个手势;③ 比例 / 旋转切换后 `offset` 是保留并重夹,还是直接归 0;④ 是否提供双击复位、灵敏度是否要做(放大越深拖得越细)。
